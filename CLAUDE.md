Python-based project. Includes notebooks for exploration and analysis, and a
full testing suite spanning unit, integration, and end-to-end coverage.

## Engineering Standards

---

### Diagrams (Required)

Every phase that introduces a new service, API endpoint, or data transformation
must include a Mermaid diagram committed to `docs/diagrams/` before the task
is marked complete.

- Use sequence diagrams for request/response and service-to-service flows
- Use flowcharts for data transformation pipelines
- Naming convention: `{phase}-{feature}-flow.md`

---

### Commit Hygiene

**Signing (Required)**

- Every commit must be GPG-signed: `git commit -S -m "..."`
- Configure globally: `git config --global commit.gpgsign true`
- Unsigned commits must not be merged; enforce via branch protection rules
- If a signing key is not yet configured, set one up before the first commit:
  `gpg --full-generate-key` then `git config --global user.signingkey <KEY_ID>`

**Message format**

- Commit messages must follow conventional commits: `type(scope): description`
  - Types: `feat`, `fix`, `docs`, `test`, `refactor`, `chore`
  - Description is written in imperative mood: "add validation" not "added validation"
- Never reference AI tools, assistants, or code generators in commit messages
- Never include phrases like "generated by", "co-authored by AI", "Claude", or
  "ChatGPT" in any commit message, body, or trailer
- Commits represent the author's work and intent; attribution belongs in docs, not git history

**Scope**

- One logical change per commit, no exceptions
- Never bundle unrelated changes in a single commit
- If a task requires multiple logical changes, split into multiple commits

---

### Linting and Formatting (Required)

All code must pass linting and formatting checks before a commit is accepted.
This is enforced via pre-commit hooks — no exceptions, no manual bypasses.

**Toolchain**

- `ruff` — linting (replaces flake8, isort, pyupgrade, and more)
- `black` — formatting (non-negotiable style, zero config)
- `mypy` — static type checking
- `bandit` — security linting (flags common vulnerabilities)
- `semgrep` — static analysis across all code including notebooks (SAST)
- `snyk` — open-source SCA, SAST, SBOM, and AI-BOM generation
- `pre-commit` — orchestrates all hooks on every commit

**Pre-commit setup**

Install hooks after cloning:

```bash
pip install pre-commit
pre-commit install
pre-commit install --hook-type commit-msg
```

`.pre-commit-config.yaml` (commit this to the repo):

```yaml
repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.5.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
      - id: check-added-large-files
      - id: detect-private-key

  - repo: https://github.com/psf/black
    rev: 24.3.0
    hooks:
      - id: black

  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.3.0
    hooks:
      - id: ruff
        args: [--fix]

  - repo: https://github.com/pre-commit/mirrors-mypy
    rev: v1.9.0
    hooks:
      - id: mypy
        additional_dependencies: [types-all]

  - repo: https://github.com/PyCQA/bandit
    rev: 1.7.8
    hooks:
      - id: bandit
        args: ['-c', 'pyproject.toml']

  - repo: local
    hooks:
      - id: snyk-test
        name: Snyk open-source vulnerability scan
        entry: snyk test
        language: system
        pass_filenames: false
        always_run: true
        args: ['--severity-threshold=high', '--fail-on=all']

  - repo: https://github.com/returntocorp/semgrep
    rev: v1.72.0
    hooks:
      - id: semgrep
        args:
          - --config=p/python
          - --config=p/secrets
          - --config=p/owasp-top-ten
          - --config=p/bandit
          - --error
          - --exclude=tests
          - --exclude=notebooks/exploratory
        pass_filenames: false
```

**Semgrep scope**

- Runs on all code in `src/` and `scripts/` on every commit
- Also runs on `notebooks/reports/` since those are production-grade artifacts
- `notebooks/exploratory/` is excluded — scratch work, not subject to SAST gates
- Rulesets applied:
  - `p/python` — general Python correctness and anti-patterns
  - `p/secrets` — hardcoded credentials, tokens, and keys
  - `p/owasp-top-ten` — injection, broken auth, insecure deserialization, etc.
  - `p/bandit` — overlaps with bandit but catches patterns bandit misses
- `--error` flag makes any finding a blocking failure; findings must be fixed,
  not suppressed, unless a `# nosemgrep: rule-id` comment with an inline
  justification is added
- To run manually across the full codebase:
  ```bash
  semgrep --config=p/python --config=p/secrets --config=p/owasp-top-ten \
    --config=p/bandit --error src/ scripts/ notebooks/reports/
  ```
- To run on a single file:
  ```bash
  semgrep --config=p/python --config=p/secrets src/package/module.py
  ```

**Snyk scope and commands**

Snyk requires authentication before first use:

```bash
snyk auth
```

The following commands are used at different stages of the workflow:

| Command          | Stage                 | Purpose                                                           |
| ---------------- | --------------------- | ----------------------------------------------------------------- |
| `snyk test`      | Pre-commit, CI        | SCA: scan open-source dependencies for CVEs and license issues    |
| `snyk code test` | CI                    | SAST: static analysis of application source code                  |
| `snyk monitor`   | CI (main branch only) | Snapshot dependencies for continuous monitoring in Snyk dashboard |
| `snyk sbom`      | CI release gate       | Generate SBOM in CycloneDX or SPDX format                         |
| `snyk aibom`     | CI release gate       | Generate AI-BOM for Python projects using AI models or datasets   |

**Pre-commit (blocking):** `snyk test` runs on every commit. High and critical
severity findings block the commit. Medium and low findings are reported but
not blocking at this stage.

**CI gates (blocking):** `snyk code test` and `snyk sbom` run in CI on every
PR. `snyk monitor` runs only on merges to the main branch to avoid duplicate
snapshots in the Snyk dashboard.

**AI-BOM:** Run `snyk aibom` on any phase that introduces or modifies AI model
dependencies, dataset references, or ML tooling. Commit the output to
`docs/aibom/` alongside the phase artifacts.

Manual commands:

```bash
# SCA: dependency vulnerability scan
snyk test --severity-threshold=high --fail-on=all

# SAST: source code static analysis
snyk code test src/

# Continuous monitoring snapshot (CI only, not local)
snyk monitor --project-name=<project>

# Generate CycloneDX SBOM
snyk sbom --format=cyclonedx1.4+json --file=pyproject.toml > docs/sbom/sbom.json

# Generate AI-BOM for Python project
snyk aibom --output=docs/aibom/aibom.json

# Inspect or update the .snyk policy file
snyk policy
snyk ignore --id=<VULN_ID> --reason="<justification>" --expiry=<date>
```

**Suppression policy**

- Vulnerabilities suppressed via `snyk ignore` must include a reason and expiry date
- The `.snyk` policy file is committed to the repo and reviewed in every PR
- Ignored findings with expired dates are treated as active findings at the next scan
- Never use `--trust-policies` in CI without explicit approval in the task summary

**pyproject.toml tool config:**

```toml
[tool.ruff]
line-length = 100
target-version = "py311"
select = ["E", "F", "W", "I", "UP", "B", "SIM"]
ignore = ["E501"]

[tool.black]
line-length = 100
target-version = ["py311"]

[tool.mypy]
python_version = "3.11"
strict = true
ignore_missing_imports = false

[tool.bandit]
exclude_dirs = ["tests", "notebooks"]
skips = ["B101"]  # Allow assert in tests
```

**Rules for executors**

- Run `pre-commit run --all-files` before marking any task complete
- Never use `git commit --no-verify` to bypass hooks
- Fix all linter errors before committing; do not suppress warnings without a
  documented reason in a code comment
- `mypy` errors are blocking; `# type: ignore` comments require an inline explanation
- Semgrep findings are blocking; `# nosemgrep: rule-id` suppressions require an
  inline explanation of why the finding is a false positive or accepted risk
- Snyk high/critical findings are blocking; use `snyk ignore` with a reason and
  expiry rather than suppressing at the CLI level

### Python Standards

**Style and formatting**

- All code must pass `ruff` and `black` as configured above
- Type hints are required on all function signatures (parameters and return types)
- Use `pathlib.Path` over `os.path` for all filesystem operations
- Prefer explicit imports over wildcard imports (`from module import *` is banned)

**Dependencies**

- All dependencies declared in `pyproject.toml` under `[project.dependencies]`
- Dev-only dependencies (pytest, ruff, black, pre-commit) under
  `[project.optional-dependencies.dev]`
- Pin versions for production dependencies; use ranges for dev
- Do not add a dependency if the standard library covers the use case

**Environment and configuration**

- All configuration loaded from environment variables via `src/{package}/config.py`
- Never hardcode environment-specific values in source code
- Use `.env.example` as the canonical list of required variables; never commit `.env`

**Module and package conventions**

- One responsibility per module; split if a file exceeds ~300 lines of logic
- `__init__.py` exposes only the public API of each package; internals stay internal
- Avoid circular imports; if they appear, it signals a layering problem

### Notebooks

Notebooks live in `notebooks/` and serve two distinct purposes:

**`notebooks/exploratory/`**

- Scratch space for analysis and experimentation
- Not required to be clean or reproducible
- Never imported by application code
- Clear outputs before committing unless the output is the artifact

**`notebooks/reports/`**

- Must be fully reproducible from top to bottom with a fresh kernel
- Outputs must be committed alongside the notebook
- Use clear markdown cells to explain reasoning between code cells
- Parameterize notebooks using `papermill` if they need to run programmatically
- No hardcoded file paths; use relative paths from the project root

**Both types:**

- Never import from `notebooks/` in `src/`
- Use `%autoreload 2` only in exploratory work, not reports
- Document the purpose of the notebook in the first markdown cell

### Security (Required)

Every task that introduces input handling, authentication, authorization, or
external communication must address the following before being marked complete.

**Input handling**

- Validate and sanitize all external input at the boundary
- Never trust data from request bodies, headers, query params, or env without validation
- Reject unexpected fields rather than ignoring them

**Authentication and authorization**

- No hardcoded credentials, tokens, or secrets in code or config files
- Secrets are loaded from environment variables or a secrets manager only
- Every protected route or resource must verify identity and permission explicitly

**Dependencies**

- Do not introduce new dependencies without documenting why in the task summary
- Prefer well-maintained, minimal-footprint libraries

**Error handling**

- Never expose stack traces, internal paths, or system details in error responses
- Log errors server-side with context; return generic messages to callers

**Least privilege**

- Functions, services, and modules request only the permissions they need
- No broad wildcard permissions in IAM policies, roles, or service accounts

---

### Code Comments

Comments explain why, not what. The code explains what.

- Add a comment when the reasoning behind a decision is not obvious from reading the code
- Document all public functions, classes, and modules with a Google-style docstring:
  function purpose, Args, Returns, Raises, and any notable side effects
- Mark intentional workarounds or known limitations with `# TODO:` or `# HACK:`
  and a one-line explanation
- Remove commented-out code before committing; use git history for recovery

**Docstring format:**

```python
def process_records(records: list[dict], threshold: float) -> list[dict]:
    """Filter records exceeding the given threshold.

    Args:
        records: List of record dicts, each containing a 'value' key.
        threshold: Minimum value a record must exceed to be included.

    Returns:
        Filtered list of records where value > threshold.

    Raises:
        ValueError: If threshold is negative.
    """
```

### Testing

Every task that introduces or modifies logic must include tests committed in the
same task. Tests are not a separate phase.

**Common rules across all tiers**

- A task is not complete if its tests do not pass
- Test files mirror the `src/` structure under `tests/`
- Use `pytest` as the test runner; configure in `pyproject.toml`
- Shared fixtures live in `tests/conftest.py`; scope them appropriately
  (`function`, `module`, `session`)
- Do not write tests that only assert the obvious
- Use `pytest-cov` to measure coverage; fail the build below 80% on `src/`

**Unit tests (`tests/unit/`)**

- Cover the core logic of every new function or class
- Test the happy path, expected failure cases, and boundary conditions
- Mock external dependencies (databases, APIs, file system) using `pytest-mock`
  or `unittest.mock`
- Must be fast and deterministic with no side effects or I/O

**Integration tests (`tests/integration/`)**

- Cover interactions between components: service-to-service, module-to-database,
  handler-to-middleware
- Use real or containerized dependencies where practical (e.g., `pytest-docker`
  for a test database)
- Verify that contracts between components are honored end-to-end within the system
- Acceptable to be slower than unit tests; isolate with `pytest -m integration`

**End-to-end tests (`tests/e2e/`)**

- Cover critical flows from entry point to final output
- Run against a deployed or locally running environment, not mocks
- Focus on the paths that matter most: primary success flow, primary failure flow,
  and any flow with security or data integrity implications
- E2E tests confirm the system works as a whole; they do not replace unit or
  integration coverage
- Mark with `pytest -m e2e` and exclude from default local runs

**Pytest markers** (declare in `pyproject.toml`):

```toml
[tool.pytest.ini_options]
markers = [
  "unit: fast, isolated, no I/O",
  "integration: requires real or containerized deps",
  "e2e: full system, deployed environment",
]
addopts = "--strict-markers -ra"
```
